## 1. Problem + Scope (locked)

harnessed hardcodes Claude Code's path layout (`~/.claude/` + sibling `~/.claude.json`). v9.0 abstracts harnessed's OWN paths onto a **platform descriptor + auto-detect**, proving portability by wiring the industry `.agents/` convention as a real second platform. Upstreams (gstack/ECC/superpowers/codegraph) are already cross-harness (v8.0 verified) ‚Üí **no generator, no upstream-mapping, no per-platform install matrix** (that was scope C, rejected as over-build).

**Scope B (D2):** PlatformDescriptor (5 path families) + `detectPlatform()` (CC default, zero behavior change) + central config resolvers + real `.agents/` wiring as the second implementation + other harnesses (Cursor/Gemini/Copilot) as descriptor **data** (not tested install targets).

## 3. Target architecture

### 3a. PlatformDescriptor (one struct, the seam)

```
interface PlatformDescriptor {
  id:              'claude' | 'agents' | 'cursor' | 'gemini' | 'copilot'
  homeDir:         string   // ~/.claude   |  ~/.agents   |  ~/.cursor
  stateRoot:       string   // <homeDir>/harnessed         (getHarnessedRoot target)
  settingsPath:    string   // <homeDir>/settings.json
  skillsDir:       string   // <homeDir>/skills
  commandsDir:     string   // <homeDir>/commands
  pluginsRegistry: string   // <homeDir>/plugins/installed_plugins.json
  mcpConfigPath:   string   // ~/.claude.json (CC: SIBLING, irregular ‚Äî see ¬ß7 D-A)
}
```

`mcpConfigPath` is the one non-uniform field: for Claude Code it is `~/.claude.json`, a sibling of `~/.claude/`, NOT under `homeDir`. Every other field is `homeDir`-relative. This irregularity is the load-bearing open question (¬ß7 D-A).

### 3b. detectPlatform() ‚Äî CC default, reversible

```
detectPlatform(): PlatformDescriptor
  1. HARNESSED_ROOT_OVERRIDE set?  ‚Üí synthetic descriptor rooted there  (tests, unchanged)
  2. HARNESSED_PLATFORM=<id> set?  ‚Üí that descriptor                    (explicit user pin)
  3. pin file <stateRoot>/.platform present? ‚Üí that descriptor          (setup --platform writes it)
  4. auto-detect, precedence:  ~/.claude/ exists ‚Üí claude  (INCUMBENT, wins)
                               else ~/.agents/ ‚Üí agents
                               else ~/.cursor/ ‚Ä¶ 
  5. fallback ‚Üí claude
```

Precedence rule **claude-first** = existing users get byte-identical behavior (reversibility / blast-radius zero). A user only lands on `.agents/` if they have no `~/.claude/`, or explicitly opt in via `HARNESSED_PLATFORM` / `setup --platform agents`.

### 3d. Migration

- `migrateLegacyHarnessedRoot()` (`~/.harnessed` ‚Üí `~/.claude/harnessed`) **stays unchanged**.
- v9.0 does **NOT** auto-migrate state across platforms. Active platform = detected platform; switching platforms = explicit re-`setup`. Documented, not silent. (Avoids a cross-platform state-merge hazard for a rare action.)

## 4. Phasing (within the v9.0 milestone)

```
Phase A ‚Äî seam (Tier 1, TDD)
  PlatformDescriptor type + detectPlatform() + route getHarnessedRoot through it.
  HARNESSED_ROOT_OVERRIDE still wins. Zero behavior change (claude default).
  ‚úÖ green gate proves no regression before touching config dirs.

Phase B ‚Äî central config resolvers (Tier 2, TDD)
  getSettingsPath/getSkillsDir/getCommandsDir/getPluginsRegistry/getMcpConfigPath.
  Swap the ~7 scattered call sites. Fold the 3 settings.json writers behind one
  path + a deep-merge atomic writer (¬ß7 D-C). DRY win + concurrent-write safety.

Phase C ‚Äî .agents/ proof (second implementation, TDD + dual-platform integration)
  agents descriptor + `setup --platform agents` + .platform pin. Verify .agents/
  internal layout mirrors .claude/ (¬ß7 D-B) BEFORE detailing. Parameterize the
  hardcoded fixture manifests. Dual-platform setup integration test.
```

Make-the-change-easy-then-make-the-easy-change (Beck): A+B are pure refactor (seam + resolvers, no new platform), C is the behavioral add (second platform). Never structural+behavioral in one phase.

## 6. Eng-review cognitive-pattern check

- **Boring by default** ‚Äî descriptor struct + detect function are proven/boring. No innovation token spent. ‚úÖ
- **Blast radius** ‚Äî claude-first precedence = zero blast on existing users; the change is a no-op until a user opts into another platform. ‚úÖ
- **Incremental** ‚Äî A/B refactor-only, C behavioral. Reversible (default unchanged). ‚úÖ
- **Essential vs accidental (Brooks)** ‚Äî the 5 path families are essential coupling; the descriptor removes accidental scatter. NOT solving a self-created problem. ‚úÖ
- **DRY** ‚Äî folds 3 independent settings.json writers into one resolver + atomic writer (a real repetition the map flagged). ‚úÖ
- **Systems over heroes** ‚Äî central resolvers mean a future 6th platform is data, not a code hunt. ‚úÖ
- **YAGNI guard** ‚Äî scope B mandates a second real implementation (`.agents/`) so the abstraction is validated, not speculative. ‚úÖ

## 7. Decisions to lock (open ‚Äî need your input)

**D-A ‚Äî `mcpConfigPath` irregularity (load-bearing).** CC's MCP registry is `~/.claude.json`, a SIBLING of `~/.claude/`, not under `homeDir`. For `.agents/`, where does MCP config live? Recommendation: make `mcpConfigPath` an explicit per-descriptor field (not derived from `homeDir`), and for `agents` verify the real convention at Phase C start before hardcoding. **Risk if wrong:** MCP install silently writes to the wrong file on `.agents/`. Needs verification, not a guess.

**D-B ‚Äî `.agents/` internal layout compatibility.** Does `.agents/` use the SAME `skills/<name>/SKILL.md` + `settings.json` + `commands/` layout as `.claude/`? Evidence it does: `idempotent.ts:120-128` already probes `~/.agents/skills/<name>/SKILL.md` (same shape). Recommendation: treat layout as compatible (descriptor = path swap only, no format adapter) ‚Äî keeps scope B minimal. If Phase C finds a schema/format divergence, that becomes a flagged scope bump, NOT silently absorbed.

**D-C ‚Äî settings.json centralization scope.** The map flagged 3 independent settings.json writers (Agent Teams, lang, hooks) ‚Äî a pre-existing concurrent-write + DRY smell. Recommendation: bundle a `getSettingsPath()` + unified deep-merge atomic writer into Phase B (the backups-to-`harnessed/backups/` pattern is precedent). It is in the blast radius anyway. **Alternative:** leave the 3 writers, only add the path resolver (smaller diff, leaves the write hazard). Rec = bundle.

**D-D ‚Äî milestone structure.** This design is the architecture lock. Next: formalize v9.0 into `.planning/` as phases (A/B/C above) via the project's GSD-native path (hand-controlled, per host overreach lesson), or keep iterating the design first. Rec: lock A/B/C as 3 phases, start Phase A with TDD.
