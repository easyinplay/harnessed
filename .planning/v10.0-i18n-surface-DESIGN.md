## Business purpose

Extend harnessed's i18n from the already-shipped **CLI message layer** (`src/i18n/index.ts` + `messages/{en,zh-Hans}.json`, `t()` loader, en-default byte-identical, shipped v3.4.0) to the **skill/workflow surface** — the orchestration brain's prompt library that gets injected/surfaced to the user. Comet model: ship EN+中文 skills as a bilingual product surface.

## Scope target (grounded in code, 2026-06-24)

- **28 `SKILL.md`** prompt bodies under `workflows/**/` — ~12,332 words total.
- **48 workflow/discipline/judgment `.yaml`** carrying user-facing `description`/prompt strings.
- Current CLI table gap: `messages/zh-Hans.json` (80 keys) trails `messages/en.json` (94 keys) — 14 untranslated keys.

## Premise challenge + decision (office-hours diagnostic)

office-hours challenged the premise directly (anti-sycophancy):

- **Challenge**: harnessed has ~0 external stars (vs comet 1.5k / Trellis 10.9k); sole operator is bilingual and drives English-named tooling end-to-end; no observed user is blocked by English skill bodies. Bilingual-izing 12k words imposes a dual-maintenance tax on every prompt edit for a user base that does not yet exist. The recommended posture was **narrow** (finish 14 CLI keys + a thin runtime 中文-output discipline, defer full translation until a zh-user signal).
- **User decision (overrides recommendation)**: **full v10.0 scope** — bilingual the entire skill/workflow surface now. Logged as a defended-premise founder signal; not re-litigated. The user carries context (intent toward a Chinese audience / value of the bilingual surface) the diagnostic does not.

## Chosen implementation approach — A: Parallel sibling files (comet model)

`SKILL.md` (en) + `SKILL.zh-Hans.md` sibling files; install/resolve layer picks by locale; a CI **sync-guard** keeps the pair from drifting.

**Rationale**: literal alignment with comet's "ship EN+中文 skills"; whole-file translation preserves prompt semantics (vs string-table slicing that mangles 12k words of prose); reuses the v9.0 platform-aware resolve layer (`detectPlatform` / resolvers) to select the file; the sync-guard turns the dual-maintenance tax into a CI-checkable hard constraint instead of silent drift.

**Rejected**: B (externalized string-table extending `t()`) — XL effort / High risk; prose prompts are not string-table shaped, slicing destroys readability, anti-karpathy-simplicity. C (build-time split single-file) — M effort but bloats source files and adds a build-time splitter + tests.

## Scope boundary

**In**: 28 SKILL.md → +28 `.zh-Hans.md` siblings; user-facing strings in the 48 yaml; locale-aware resolve for the skill/workflow surface; CI sync-guard (en/zh pair parity); close the 14-key CLI table gap (80→94).

**Out / deferred**: third language beyond en + zh-Hans (forward-compat hook only, per existing `mapToSupported`); translating internal-only yaml that never surfaces to a user; runtime LLM auto-translation (static files only).

## Risks / tradeoffs

- **Dual-maintenance drift** — primary risk; mitigated by the CI sync-guard (hard gate, not advisory).
- **File count doubling** (28→56 SKILL.md + yaml siblings) — accepted; mirrors the messages/{en,zh} file-level pattern.
- **Premature for current user base** — accepted by user override; the bilingual surface is a forward bet, not a response to observed demand.

## Milestone sketch (粗粒度 — GSD restructures)

1. **Resolve layer**: locale-aware sibling-file selection for skill/workflow surface (extend v9.0 resolvers).
2. **Sync-guard**: CI check enforcing en ↔ zh-Hans pair parity (presence + structural drift).
3. **Translation**: 28 SKILL.md → zh-Hans siblings + surfacing yaml strings (TDD where mechanism, not prose).
4. **CLI gap close**: 14 untranslated keys (80→94).
