## Implementation Decisions

### D-01: checkpoint 摘要生成策略 = TEMPLATE (机械 extract 固定 fields, zero LLM call)

**Selected**: TEMPLATE — 机械 extract 固定 fields, zero LLM call

**Locked rationale**: Karpathy YAGNI 原则 + R7.2 验收"单 checkpoint < 1k token"明显指向机械拼装不是 LLM. zero API 依赖 / deterministic / 极快 / 可纯代码验证 — 这些都是 infra 底座必备品. LLM-based summarization 引入 cost / 不 deterministic / 可能超 1k token 需 retry, 是 over-engineering. 表达力受限于预设 field 是接受的代价 (checkpoint 不是 conversation 重建, 是 next-phase reload 用 minimal context).

**Implementation hint**:
- `src/checkpoint/template.ts` ~60-80L NEW — 固定 schema: `{ schemaVersion: 'checkpoint.v1', phase, status: 'active'|'paused'|'complete', last_task: string, key_decisions: string[], canonical_refs: string[], session_id?: string, timestamp: ISO, archive_path: string }`
- TypeBox schema validate (sister Phase 2.2 schemaVersion 7-surface pattern: `branchOnSchemaVersion<T>(v, { 'checkpoint.v1': handler })`)
- write 时机: 每 phase 完成 + session paused signal (SIGINT trap OR `harnessed compact` 触发)
- 摘要 < 1k token 验证: 用 `Buffer.byteLength(JSON.stringify(checkpoint), 'utf8') / 4` 估算 (1 char ≈ 0.25 token Heuristic); 超过则 fail-loud (truncate `last_task` 等 string field)
- archive 路径独立: `.harnessed/archive/<phase>/raw-{timestamp}.json` 写完整 turn 历史 (不进 context, 仅供回溯)

**decision_source**: 用户首轮决策 TEMPLATE (Recommended option)

---

### D-02: `current-workflow.json` 状态机模型 = KARPATHY 极简 3 state

**Selected**: KARPATHY — `active` / `paused` / `complete` 3 state

**Locked rationale**: YAGNI 避免 over-modeled FSM. 3 state 充分表达 "在跑 / 中断了 / 跑完了" 三种实际语义. resume 逻辑极简: `if status==='paused' load checkpoint else noop`. 6-state FSM (idle/discussing/planning/executing/awaiting-checkpoint/recovering) 是 textbook 完整但 phase 3.1 MVP 不需要也不能测所有转移 (discuss/plan/execute 本身是 phase 顶层, 不是 checkpoint state). 字符串 enum 比 nested object 简单 50%.

**Implementation hint**:
- `.harnessed/current-workflow.json` schema: `{ schemaVersion: 'current-workflow.v1', phase: string, status: 'active'|'paused'|'complete', last_checkpoint_path: string, started_at: ISO, paused_at?: ISO, completed_at?: ISO }`
- write 时机: phase 启动写 `active` + last_checkpoint_path=null; checkpoint 写入时同步更新 last_checkpoint_path; SIGINT trap 写 `paused` + paused_at; phase 完成写 `complete` + completed_at
- 不引入额外 state library (xstate / robot3) — 1 个 enum field + 几个 if 足矣
- Karpathy hard limit `src/checkpoint/state.ts` ≤80L

**decision_source**: 用户首轮决策 KARPATHY (Recommended option)

---

### D-03: `harnessed resume` 行为 = RELOAD (reload checkpoint 进 context, 用户手动续跑)

**Selected**: RELOAD — `harnessed resume` 输出 checkpoint 摘要 + 提示用户手动调起 phase command

**Locked rationale**: Claude Code 交互底色 = 用户总需手动 invoke skill (slash command). resume 不偷袭 = 用户保留"忘记重开"自由 (有时确实想跳到别的 phase). AUTO-REPLAY 需设计可靠 command-rehydration 机制 (Claude Code 环境外不一定能调起 slash command), 决定路径不明造成 unexpected 重跑. RELOAD 多报一行命令的代价远低于 AUTO-REPLAY 黑盒风险. 与 R7.3 验收"resume 不丢上下文"完全兼容 (用户看到 checkpoint 摘要即可"不丢上下文").

**Implementation hint**:
- `src/cli/resume.ts` ~50-70L NEW — CLI subcommand `harnessed resume`
- 读 `.harnessed/current-workflow.json` → 找 status==='paused' phase
- 读 last_checkpoint_path → JSON parse + validate (TypeBox checkpoint.v1)
- 输出 (stdout, 不写盘): checkpoint 摘要 (phase / last_task / key_decisions 5-10 行) + "→ 在 Claude Code 内调用 `/gsd-execute-phase <X.Y>` 续跑 (session_id 已捕获, SDK 会 redirect 续原会话)"
- session_id 不存在时 fallback: "→ 在 Claude Code 内调用 `/gsd-execute-phase <X.Y>` (fresh session, context 已 reload)"
- exit code 0 = pass, 1 = no paused phase found / file corrupt / schema 不匹配 (fail loud)
- `--json` flag CI 友好输出 (sister doctor.ts pattern)

**decision_source**: 用户首轮决策 RELOAD (Recommended option)

---

### D-04: T4.4 Task Session closure infra wire-in 决断 = WIRE-IN

**Selected**: WIRE-IN — `session_id` 写 checkpoint, resume 时 SDK redirect 续原 session

**Locked rationale**: Phase 2.2 T4.4 三件套 closure infra 已 ready 等了 1 milestone (`sdkSpawn.onSessionId` / `ralphLoopWrap.resumeSessionId` / `engine.wrappedSpawn capturedSessionId`), defer 进 v0.3.0 后 backlog 又 1 milestone 闲置就成 dead infra. WIRE-IN 实现成本极低 (3 件套已 closure 安, 仅 checkpoint schema 加 `session_id?` field + resume 调 SDK 时传入). 业务价值大: SDK session 续跑 = lossless conversation context 恢复 (比 reload checkpoint 摘要 80% 损耗信息密度高很多). 与 R7.3 验收"resume 不丢上下文"匹配度最高 (机器层"session 续跑"比文档层"checkpoint reload"信息更完整).

**Implementation hint**:
- `src/checkpoint/template.ts` schema 加 `session_id?: string` (optional — 兼容 fresh-session phase 不强求)
- phase 启动时通过 `sdkSpawn.onSessionId(cb)` callback 捕获 SDK session_id → 写入下一次 checkpoint
- `harnessed resume` 输出含 session_id 时 prompt 用户 (D-03 RELOAD 流程): "session 可续跑 (sid: xxx)"
- 用户在 Claude Code 内调起 phase command 后, executor 检 `current-workflow.json` last_checkpoint.session_id 存在 → 走 `ralphLoopWrap.resumeSessionId(sid)` SDK path
- fallback: session_id 不存在 / SDK 拒绝续 (session 过期 etc) → fresh session + reload checkpoint 摘要 (退化路径无破坏)
- Phase 2.2 closure infra 3 件套都已 ready, 本 phase 仅"加 schema field + wire 接入"~20-30L 工作量
- 测试: integration test 起 SDK fake session → SIGINT 模拟中断 → 跑 `harnessed resume` → 验 session_id 透传 + SDK redirect 成功 (mock SDK)

**decision_source**: 用户首轮决策 WIRE-IN (Recommended option)

## Specific Ideas

- **D-01 TEMPLATE schema 灵感**: `{ phase: '3.1', status: 'paused', last_task: 'W2 T2.2 SDK session_id 捕获', key_decisions: ['D-01 TEMPLATE', 'D-04 WIRE-IN'], canonical_refs: ['ROADMAP.md L156-158'], session_id: 'sess_abc123', timestamp: '2026-05-16T10:30:00Z', archive_path: '.harnessed/archive/phase-3.1/raw-2026-05-16T10-30-00Z.json' }` — 实际 size ~400 chars ≈ 100 tokens, 余 900 token headroom 给 `last_task` + `key_decisions` 扩展.
