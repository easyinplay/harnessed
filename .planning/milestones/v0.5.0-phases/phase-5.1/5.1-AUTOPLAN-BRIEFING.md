## R10.1 audit log --filter CLI consumer (新功能)

### Scope

`harnessed audit log --filter <expr>` CLI subcommand consumer (Phase 4.3 R8.1 已 ship JSONL producer `.harnessed/audit.log` 12-field schema)。

### 核心议题 (gstack 多角色 stress-test)

**Q1 (Designer / Paranoid Staff Eng)**: filter expression syntax 选择 — jq pipe (`jq` runtime dep 但 doctor 3rd check 已 verify 装齐) vs 内置 JSONPath (`jsonpath-plus` npm dep ~30KB) vs simple field=value DSL (zero-dep 但表达力弱)?

- jq pipe: 复用现有 jq dep (zero-new-dep), 工程师熟悉,表达力极强 `jq 'select(.category=="engineering")'`;但需要把 audit.log JSONL pipe 给 jq,UX 是 `cat .harnessed/audit.log | jq ...` 还是 `harnessed audit log --filter '.category=="engineering"'` 内嵌? **推荐内嵌 jq invocation,UX 一致 + 不暴露文件路径**
- JSONPath: npm dep 维护负担, 但 Python 工程师 familiar;harnessed 受众主要 TS/JS 工程师,jq 更对路
- DSL: 自写 parser 维护负担最大, 表达力最弱 (无 OR / AND 嵌套), Karpathy YAGNI

**Q2 (EM / QA)**: output format default — JSON pretty-print vs human-readable table vs --json/--table flag both?

- 默认 human table (12 字段 condense 显 5-6 关键列: ts, phase, category, matched_rule_id, outcome, iter_count);`--json` flag 完整 12 字段 pipe-friendly
- 推荐 dual format,human default + pipe-friendly opt-in (`--json`)

**Q3 (Paranoid Staff Eng)**: pagination + sort flag surface (Karpathy YAGNI vs UX completeness):

- 最小 MVP: `--tail N` (recent N) + `--head N` + `--reverse`
- 完整: `--tail N` + `--head N` + `--since <ISO>` + `--until <ISO>` + `--reverse` + `--sort <field>`
- 推荐最小 MVP (3 flag), --since/--until/--sort 推 v0.6+ if signal

**Q4 (CSO security)**: audit.log 含 `task_excerpt` (truncated 200 char) — 是否需要 redact 敏感 patterns (API key / token / password) before display in CLI consumer?

- 推荐: 在 audit producer (Phase 4.3 src/audit/log.ts) 入口已 sanitize task_excerpt,consumer 直读 OK;但 R10.1 验收必须 verify producer 端 sanitize coverage
- 边界情况: 用户 task 含 `--token=abc123` 字面 → 已存进 audit.log → consumer 显示 = 信息泄露;建议 consumer 端 ALSO 加 secondary redact pattern (defense-in-depth)

## R10.2 src/checkpoint/state.ts concurrent write lock (#BU)

### Scope

`writeCurrentWorkflow` + `engineHook.activatePhase` / `completePhase` 直接 `writeFile` 无锁;并发 `harnessed resume × 2` 可竞态损坏 current-workflow.json。

### 核心议题 (gstack 多角色 stress-test)

**Q5 (Paranoid Staff Eng / Eng Manager)**: lock implementation 选择 — `proper-lockfile` npm dep (5M weekly downloads battle-tested) vs 手写 `fs.open(wx)` exclusive create + retry (zero-dep 但 race window) vs OS flock (cross-OS Win 不支持)?

- proper-lockfile: 6 KB minified, no runtime dep, MIT license, cross-OS,Windows ACL workaround 已知,5M weekly downloads = bus factor 安全
- 手写 `wx`: race window minimal but exists (open with `wx` returns EEXIST → handle → retry → re-check),且 cleanup 必须考虑 process crash leftover lock file
- OS flock: Win 不支持 → 直接淘汰

**推荐**: `proper-lockfile` (引入 1 个 dep 换取 robust + cross-OS;sister CLAUDE.md Win corepack ACL workaround 教训 — 手写 cross-OS file lock 是 footgun)

**Q6 (QA)**: retry policy — bounded retry (3 × 100ms exp backoff) + clear error / unbounded retry (wait forever) / fail-fast (NO retry throw immediately)?

- fail-fast: 友好 UX 弱 (用户碰到 lock 就 fail)
- unbounded: 死锁风险 (上一个 resume process 崩了 leftover lock)
- bounded retry: 3 × 100ms exp backoff (~700ms total) 然后 throw — UI 友好 + 死锁 bounded

**推荐**: bounded retry 3× exp backoff + clear error message hint "another harnessed process holds the lock — wait or kill stale process"

**Q7 (Designer)**: stale lock recovery UX — 是否提供 `harnessed resume --force` flag bypass lock?

- 不提供: 用户必须找到 stale process kill 它,UX 差
- 提供 --force: 但 race condition 可能 lock 被另一个真活着的 process 持有,--force 会损坏 state

**推荐**: 不提供 --force,但提供 `harnessed status` 显示 lock holder pid + timestamp;stale 判定靠 `proper-lockfile` 自带 stale 检测 (default 10s)

**Q8 (CSO security / Paranoid Staff Eng)**: 锁的范围 — 整个 `.harnessed/` dir vs 单文件 `.harnessed/current-workflow.json` vs operation-level lock?

- 单文件 lock: minimal surface,但只保护 writeCurrentWorkflow;engineHook activatePhase/completePhase 写其他 checkpoint files 还是会 race
- dir lock: 保护范围最大,但锁粒度粗
- operation-level: 每个 fn 自己锁自己写的文件,锁粒度精细但实现复杂

**推荐**: dir 级 lock (`.harnessed/.lock`) 由 proper-lockfile 管理,粒度粗但简单 + 安全 (Phase 5.1 NEW infra YAGNI,未来 v0.6+ 有信号再 refine)
