## Cross-Phase Integration

The three phases compose into one per-turn surface (each builds on the prior):

- **34 → 35**: Phase 34 writes the active workflow to a session-composite slot; Phase 35's bin became session-aware to READ that slot (else it would inject the wrong/last session's state via the legacy mirror under concurrency). The cross-harness seam (`sessionIdEnv`) introduced in 35 retro-fixed 34's hardcode — one resolution model, not two.
- **35 → 36**: Phase 35 wired the per-turn breadcrumb hook; Phase 36's `VERIFY-MODE` line rides that exact surface, so the scale-adaptive directive actually reaches the agent each turn instead of sitting unread on the envelope.
- Proven by: 24/24 + 25/25 + 29/29 isolated, full serialized 1470/0, and three PowerShell real-binary e2es (composite-key write; two-session inject isolation; VERIFY-MODE full/light/absent).

No blocker-class gaps. Three non-blocker follow-ons recorded as tech debt (bin minimal platform-resolve; codex sessionIdEnv null placeholder; perturn-inject opt-in not default).

## End-to-End Flows

1. **single-session incumbent (zero blast)** — with no session id (scripts, CI, single CC session), `activeKey()===repoKey()`, the inject bin reads the bare slot, and a workflow with no `verify_mode` emits the same breadcrumb as before. Byte-identical to pre-v11.0. Proven: all pre-existing state-store/inject cells green unchanged + the absent-field assertions.
2. **concurrent-session isolation** — two CC sessions in one repo (A=`task`, B=`verify`) seeded via the real CLI hold two coexisting `<repoKey>::<sid>` slots; the session-aware bin injects each session's OWN workflow-state (+ its scale-adaptive VERIFY-MODE directive when set), reading the composite slot not the legacy mirror. Proven: the two-session PowerShell e2e (A→phase task, B→phase verify).

## Milestone Decisions (carried forward — durable rationale)

Preserved here at close (the ROADMAP active block collapses to a one-line shipped-index row):

1. **Strategy gate skipped (transparent)** — Spec 2/3 scope was pre-defined in the v5.0 design doc; office-hours/ceo-review skipped as not a fresh product direction, declared at milestone-open. Not re-litigated.
2. **Composite key, not `sessions/<id>.json`** — the v5.0 sketch predated Phase 15's repo-keyed `workflows.json` store; a composite map-key overlays the session axis with zero schema change + one write/lock path (rejected separate-file SoT + nested-map schema bump).
3. **Cross-harness session seam (user-flagged)** — `CLAUDE_CODE_SESSION_ID` is CC-specific; routed through `PlatformDescriptor.sessionIdEnv` so new harnesses add one descriptor field, no call-site churn — and Phase 34's hardcode was de-hardcoded onto the same seam in-milestone.
4. **Feasibility self-resolved (no eng-review)** — `ccHookAdd` already supports `UserPromptSubmit` and a SessionStart hook (`dashboard-autospawn`) already ships, so Phase 35 needed no `/plan-eng-review` despite the v5.0 sketch flagging it.
5. **Surface verify_mode in the breadcrumb** — chosen over a verify-prompt injection: verify_mode IS workflow-state, the breadcrumb is the per-turn surface Phase 35 wired, and the SHIP-READY/RETRO-DUE precedent made it a one-line zero-new-mechanism change.
