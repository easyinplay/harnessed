## 4. Setup pipeline (src/cli/setup.ts)

`setup.ts` is 276 LOC. Action handler runs 6 sequential steps (A → A.5 → A.6 → C → D → B). `--dry-run` flag opt-in; default is apply-immediate (v3.0.1 UX flip).

**Step A — Workflow skill install**
- Source: `<packageRoot>/workflows/<name>/SKILL.md` (recursive scan via `scanWorkflowsWithSkill` in `setup-helpers.ts`).
- Dest: `~/.claude/skills/<name>/` (whole directory copy via `fs.cp({recursive:true, force:true})`).
- v2-deprecated workflows (`plan-feature` / `execute-task` / `verify-work`) emit deprecation block before install summary (`renderDeprecationBlock`).
- Idempotent: `force: true` overwrites whole dir on re-run; existing user customizations under `~/.claude/skills/<workflow>/` are clobbered.

**Step A.5 — Template render (`{{ capabilities.<name>.cmd }}` substitution)**
- File: `renderAllSkills(skillNames, skillsBase, workflowsDir)` in `renderSkillTemplates.ts` (138 L).
- For each just-installed `~/.claude/skills/<name>/SKILL.md`, parses `{{ capabilities.<X>.cmd }}` placeholders via regex `/\{\{\s*capabilities\.([a-zA-Z0-9_-]+)\.cmd\s*\}\}/g`, looks up in `capabilities.yaml`, replaces with literal `cmd` string.
- Probe: `readInstalledPlugins()` (parses `~/.claude/plugins/installed_plugins.json`) + `readInstalledUserSkills()` (lists `~/.claude/skills/` dirs). When `install_type` declared and backing missing, emits warning (does NOT mutate cmd).
- v3.4.1 hotfix that this exists: pre-v3.4.1 the placeholders shipped verbatim into installed SKILL.md, so users saw literal `{{ capabilities.gstack-review.cmd }}` strings.

**Step A.6 — `~/.claude/commands/<x>.md` generation (v3.4.3)**
- File: `writeAllCommands` in `generateCommands.ts` (229 L).
- Dest: `~/.claude/commands/<slash-name>.md` (one file per role-prompt entry).
- Source: `workflows/role-prompts.yaml` × `workflows/capabilities.yaml` × installed plugin/user-skill sets.
- Body schema: `# /name` + description + `## How to invoke` (preferred path → `Use the SlashCommand tool to run <cmd>` + fallback path → `Use the Task tool to spawn a general-purpose subagent with this prompt: > role + mission + checklist + severity + output format`) + `## Notes`. Master entries get a dispatcher-only preferred path with no per-sub checklist.
- Idempotent: skips if file already exists at `<commandsDir>/<name>.md` (`fileExists(path)` returns true → log "leaving user file unchanged"). Additive only, never overwrites a user-authored file.

**Step C — Agent Teams env auto-enable**
- File: `enableAgentTeamsInSettings` in `enableAgentTeamsInSettings.ts` (107 L).
- Dest: `~/.claude/settings.json` (creates if missing; merges `env.CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS = "1"` into root-level `env:` block per Q-AUDIT-5b LOCKED schema).
- Backup: writes `<settings>.backup.<ts>` before mutation when merging into existing file.
- Idempotent: returns `'already-enabled'` if env var present with value `"1"`.

**Step D — User language preference (v3.4.0)**
- File: `enableUserLangInSettings(raw.userLang)` in `enableUserLangInSettings.ts` (153 L).
- Dest: `~/.claude/settings.json` → `env.HARNESSED_USER_LANG = 'en' | 'zh-Hans'` (CLI flag `--user-lang` overrides OS locale detection).
- Idempotent: returns `'already-set'` if env var present; otherwise creates with same backup pattern as Step C.

**Step B — Base manifest install (parallel)**
- Source: `<packageRoot>/manifests/tools/*.yaml` + `<packageRoot>/manifests/skill-packs/*.yaml` (auto-glob via `listBaseManifests`).
- Engine: `runStepBInstall(manifestPaths)` in `setup-helpers.ts` → spawns `runInstall()` from `src/installers/index.ts` in parallel via `Promise.allSettled` (v1.0.3 T1.1 ~75% speedup vs serial).
- Categorized outcomes: `installed` / `alreadyInstalled` / `skipped` / `failed`.
- Idempotent: each installer reads `~/.claude/harnessed/state.json` and is responsible for its own re-run safety. `alreadyInstalled` is the common return on second-and-later runs.

**End-of-pipeline output:** prints bundled-distribution highlight (workflows live in `<packageRoot>/workflows/`, NOT user-dir override per D-01 LOCKED).

## 7. True gap list (synthesis)

### User invoke surface → runtime disconnect

When a user types `/verify-paranoid` in Claude Code:

1. **Claude Code platform** detects the slash command, loads `~/.claude/commands/verify-paranoid.md` (written by Step A.6 of `harnessed setup`).
2. **The model reads the body** which currently contains:
   - **Preferred path:** "use the SlashCommand tool to run `/review`" — **Claude has no SlashCommand tool**. This branch is 100% dead.
   - **Fallback path:** "use the Task tool to spawn a general-purpose subagent with this prompt: > You are a **Paranoid Staff Engineer**... [checklist] ... [output format]". This works — Claude does have a Task tool and will spawn a subagent with the embedded role prompt.
3. **Sister `~/.claude/skills/verify-paranoid/SKILL.md`** carries the same dual-path text + a `trigger_phrases:` block. The model may match those phrases organically (without the user typing `/verify-paranoid`) and follow the same dual-path body.

**What does NOT run end-to-end:**
- `runMasterOrchestrator` is never called.
- The `delegates_to[]` in `workflows/verify/auto/workflow.yaml` (8-sub fan-out with gate eval for progress/code-review/paranoid/qa/security/design/multispec/simplify) is never honored. The user's `/verify-paranoid` invocation jumps straight to a single Task-spawned subagent.
- Any `judgments/*.yaml` gate logic (e.g. `verify-paranoid-critical.fires == phase.is_critical_module == true`) is never evaluated. The fallback subagent receives the prompt regardless of whether the module is critical.
- The 6 `disciplines/*.yaml` machine-encoded CLAUDE.md sections (karpathy / output-style / language / operational / priority / protocols) never propagate to the spawned subagent's system prompt.
- All hooks (`before-spawn` arbitrate / `after-output` style check / `before-commit` biome preempt) are dead for this invocation path.
- `capabilities.yaml` matters only insofar as Step A.5 renders `{{ capabilities.gstack-review.cmd }}` → `/review` in the body text. After install time, the file is unused at runtime.

The same disconnect holds for all 24 commands files generated by `harnessed setup`.

### 24 `commands/<x>.md` body dual-path snapshot

Every file generated by `generateCommands.ts` has identical structure:
```
---
description: "<frontmatter description from role-prompts.yaml>"
---
# /name
<description repeated as body intro>

## How to invoke

**Preferred path** (when the upstream specialist is installed): use the SlashCommand tool
to run `<resolved capability.cmd>` — the upstream specialist takes over.

**Fallback path** (when the upstream isn't installed or returns no result): use the Task tool
to spawn a general-purpose subagent with this prompt:

> You are a **<specialist>**.
> **Mission**: <responsibility>
> **Default-suspect mode**: ...
> **Review checklist**:
> 1. ...
> ...
> **Output format**: structured report ...

(Role prompt is self-contained — works even when the upstream isn't installed.)

## Notes
- Generated by harnessed setup ...
- Companion SKILL.md ...
- If you see ⚠️ ... not installed warning ...
```

**Confirmed:** the "preferred path" is vapor for every single one. The model attempts to "use the SlashCommand tool" but no such tool exists in the model's tool inventory. The only working invocation is the "fallback path" Task-spawn with embedded role prompt. Master orchestrator commands (`auto` / `discuss` / `plan` / `task` / `verify`) instead say "dispatch to per-sub slash commands" in their preferred path, which transitively also relies on the same broken SlashCommand assumption.

### Architecture options to bridge invoke surface → runtime

**Option α — Universal `harnessed run <workflow-name>` subcommand**
- One new CLI handler that takes the workflow name (e.g. `verify-paranoid`, `auto`, `discuss`, etc.), resolves the yaml path via the same logic as `resolveSubYamlPath` + `resolveMasterYamlPath`, calls `runWorkflow(yamlPath, vars, opts)`.
- `commands/<x>.md` body becomes: `Use the Bash tool to run: harnessed run <name>` + a tiny fallback role-prompt blurb for offline use.
- **Work estimate:** 3-5h. One new file `src/cli/run.ts` (~80 LOC), one register call in `cli.ts`, body template change in `generateCommands.ts` (~20 LOC), update `~10` snapshot tests.
- **Advantages (per audit):** activates the entire `src/workflow/` subsystem — `runMasterOrchestrator` + `judgmentResolver` + `disciplineLoader` + all hooks reach production. 4-stage namespace-layered architecture (ADR 0031) becomes user-visible. Minimal new code (one wrapper).
- **Disadvantages (per audit):** still depends on `_dispatchSkillStub.fn` in `run.ts:34` returning real output — currently it returns `{output: '<stub for X>'}`. A real spawn driver (SDK query / `sdkSpawn`) needs to be wired in for non-master phases or those phases will execute a stub. The master-orchestrator default `defaultSpawnDriver` recursively calls `runWorkflow`, which terminates at non-master subs that hit the stub. So Option α as a pure CLI-wire is incomplete without also wiring the dispatch stub to real `runRouting()` or SDK spawn. Adds ~1 extra layer to the Claude UX (Bash invoke wrapping a slash command).

**Option β — 13+ independent CLI subcommands**
- One CLI handler per sub-workflow (`harnessed verify-paranoid`, `harnessed discuss-strategic`, etc.). Each loads its specific yaml + invokes `runWorkflow` (or `runMasterOrchestrator` for master entries).
- Master entries (`/auto`, `/discuss`, `/plan`, `/task`, `/verify`) get their own handler each that hits the master orchestrator path.
- **Work estimate:** 1-2 days. 24 handler files (~30 LOC each = ~700 LOC scaffolding), 24 register calls, 24 unit tests, body-template change.
- **Advantages (per audit):** each subcommand has explicit `--help` text + can be independently flagged (`--max-iterations`, `--model`, etc.). Mirrors execute-task / research existing pattern.
- **Disadvantages (per audit):** large surface area for 24 near-identical handlers; violates karpathy "smallest change that satisfies acceptance" given Option α achieves the same coverage with a single handler. `cli.ts` grows from 15 to 39 register calls. Test surface bloats correspondingly. Same `_dispatchSkillStub` problem as Option α.

**Option γ — No CLI wire (accept Claude model-driven execute)**
- Leave `harnessed --help` at 15 subcommands. Rewrite the `commands/<x>.md` body template in `generateCommands.ts` to drop the broken "preferred path" entirely.
- New body schema: role + mission + checklist + acceptance + scope (per user reframe 2). Claude reads, decides whether to in-session execute / Task-spawn / Agent Team based on its routing intelligence.
- **Work estimate:** 2-3h. Body template change in `generateCommands.ts` (~50 LOC delta) + 24 file regeneration on next `setup` + matching change to SKILL.md sources (`workflows/<...>/SKILL.md` "How to invoke" sections — currently produced by `scripts/rewrite-skill-invoke-sections.mjs`).
- **Advantages (per audit):** simplest fix; removes the vapor instruction entirely; portable (works in any harness that loads markdown commands); honors user reframe 2 (don't prescribe spawn mechanism).
- **Disadvantages (per audit):** `src/workflow/` subsystem stays dead. All 6 disciplines + 10 judgments + master orchestrator + helpers + judgment resolver + discipline loader + governance + interpolate + hooks remain unreachable from production. That's ~1,400 LOC of tested but unused code (plus the schema files). Long-term either delete the subsystem (massive surface loss) or accept it as latent infra for a future Option α/β/δ.

**Option δ — Hybrid (body offers both modes; Claude chooses)**
- `commands/<x>.md` body contains BOTH the self-contained role-prompt (per Option γ) AND a one-line escape hatch: "For full runtime dispatcher (disciplines + judgments + hooks), run: `Use Bash to run: harnessed run <name>`".
- Claude decides per task context whether the in-session role-prompt is enough or whether to invoke the runtime via Bash.
- **Work estimate:** 4-6h. Combines Option α's CLI wire (~3h) with Option γ's body rewrite (~2h) + a clear decision rubric in the body for Claude.
- **Advantages (per audit):** both worlds — runtime alive (Option α benefit), simple-task lightweight path also available (Option γ benefit). User keeps optionality.
- **Disadvantages (per audit):** model behavior becomes less predictable (two valid paths with no hard rule for which to pick). Body length grows ~2×, more tokens consumed per command read. Same dispatch-stub gap as Options α and β until `_dispatchSkillStub.fn` is replaced with a real spawn driver. Decision rubric for Claude is fuzzy ("when to escape to harnessed run?"); could be debated indefinitely.

## 8. Bonus findings

1. **`_dispatchSkillStub.fn` in `src/workflow/run.ts:34` is a stub** that returns `'<stub for ${skillName}>'`. It's the only place where the workflow runtime would "actually do something". Currently no production code replaces it — every spawn through `runWorkflow` outside tests writes a placeholder string. Even Option α / β / δ that wire the CLI still need a real dispatch implementation (likely `sdkSpawn` or `runRouting`) before the runtime produces real output.

2. **`execute-task` (the only LIVE invoke path that touches workflow code) uses `loadPhases` parse-only**, not `runWorkflow`. It loads phases.yaml to extract `phase.fallback.max_iterations_exceeded` and force `phase.model='inherit'`, then hands the task off to `runRouting()` (a completely separate engine in `src/routing/`). So even the one phases-yaml-reading CLI handler bypasses the orchestrator subsystem.

3. **`scripts/rewrite-skill-invoke-sections.mjs` exists** (matched in the "SlashCommand tool" grep). It is the source of the current dual-path body content in `workflows/<...>/SKILL.md` — meaning a single script rewrite + `harnessed setup` re-run would propagate any new body schema to all 24 SKILL.md files. This significantly reduces Option γ/δ rollout cost.

4. **`workflows/judgments/fallback.yaml` uses a different schema (`rules:`) than the other 9 files (`triggers:`)**. `judgmentResolver.ts:58` dispatches on `fileName === 'fallback'` to route to `JudgmentRulesFile`. Any gate ref like `judgments.fallback.<x>.<y>` is therefore documented to always throw at the `fieldName` extraction step (line 86-88 — `fires` / `skips` fields don't exist on rules entries). This is intentional per the inline comment "fallback rules 无 fires_when/skips_when 字段, 不参与 expr eval — gate ref 指向 fallback file 时 fieldName 必走 error path". A future contributor wiring a gate to `judgments.fallback.*` would hit a TriggerNotFoundError → in `run.ts:128` the fail-soft branch kicks in (gate fires=true). Worth a comment + lint check if Option α / β go live.

5. **`workflows/auto/workflow.yaml` references `tools_available: [agent-teams-create, planning-with-files]` but the spawn driver is in-process recursive (`runWorkflow`).** Agent Teams env auto-enable (Step C of setup) writes `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` into `~/.claude/settings.json`, but no code path inside `runMasterOrchestrator` or `defaultSpawnDriver` actually invokes the Claude Code `TeamCreate` / `Agent` / `SendMessage` APIs. The `tools_available` field is decorative at runtime — only the schema validator checks its presence.

6. **`docs/WORKFLOW.md` claims the 4-stage architecture is end-to-end live** but the gap analysis section may not yet reflect the post-v3.4.0 CLI disconnect. Worth a re-read alongside any v3.4.4 spec to ensure the doc doesn't promise behavior that requires Option α/β/δ to ship.

7. **`tests/workflow/masterOrchestrator.test.ts` is 614 lines** — the largest single test file. It exercises every gate-eval / serial-parallel / arbitrate path with comprehensive fixtures. The runtime is well-tested; the only missing piece is a real production entry point. Any wire-up option above can lean heavily on these tests for regression coverage.

8. **`workflows/execute-task/phases.yaml` does NOT have a sibling `workflow.yaml`** (the directory contains only `SKILL.md` + `phases.yaml`). All other workflow directories have a `workflow.yaml`. This is the legacy v1/v2 transition — execute-task is the original ralph-loop driver that predates the v3 4-stage namespace layout. It's the one workflow whose user-invoke path (CLI subcommand → routing engine → ralph-loop) is fully wired.
