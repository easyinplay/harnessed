## GA-1: R8.1 audit.log format + scope

**Decision**: D-01 JSONL append-only + 11 字段 routing-decision schema

### Format alternatives

| Option | Pros | Cons | Selected |
|--------|------|------|----------|
| JSONL (1 line per decision) | Machine-parseable (jq) + human-readable (grep) + append-safe (no rewrite) | None significant for this use case | ✓ |
| JSON array | Standard JSON tooling | Append requires rewrite (unsafe concurrent) | |
| Plain text log | Simple, human-readable | Loses structure for future filter/dashboard | |
| Both (JSONL + plain) | Max flexibility | 维护 2x; JSONL 双兼 | |

### Scope alternatives (what to include per decision)

| Scope | Verdict |
|-------|---------|
| Routing decision ONLY (matched_rule, primary_expert, category) | Too minimal — misses outcome traceability |
| Routing decision + outcome metadata (11 字段: ts, phase, task_excerpt, task_sha1, matched_rule_id, primary/secondary_expert, category, route_layer, outcome, session_id, iter_count) | ✓ SELECTED — verifiable trace satisfies R8.1 "100% 可追溯" |
| + Full agentDef 14 字段 | ❌ REJECT — security (skill path username leak) + 体积爆炸 |
| + Subagent text/result body | ❌ REJECT — sensitive code leak + 体积 |
| + MCP secrets / env vars | ❌ REJECT — security |

**Notes**: Storage `.harnessed/audit.log` (already gitignored); single file append-only; downstream consumer = future `harnessed audit log --filter` 子命令 (v0.5+) + grep + jq one-liner
