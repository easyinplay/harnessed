## § 2 R8.1 Audit Log Technical Research

### 2.1 Hook Insertion Point Analysis

engine.ts is exactly 200L [VERIFIED: wc -l]. The A7 守恒 constraint means engine.ts must NOT grow. The audit hook MUST extract to a separate file, following the same pattern as `src/checkpoint/engineHook.ts` (49L) extracted from engine.ts at Phase 3.1.

**Single emit vs dual emit decision:**

engine.ts has two candidate points:
- L113-114: after `buildDecision(matched)` — decision known, outcome NOT yet known
- L188-189: after `await ralphLoopWrap(wrappedSpawn, maxIter)` + `await completePhase(...)` — outcome known, all 11 fields available

Single emit at L188-189 (outcome point) is the correct choice:
- Atomic write: all 11 fields available simultaneously (including `outcome`, `iter_count`, `session_id`)
- No partial records: a dual-emit (decision + outcome) would create orphaned decision records if spawn fails mid-way
- `iter_count` is only available from `ralphLoopWrap` result
- `session_id` (`capturedSessionId`) is populated by the spawn closure (L183-185)
- `outcome` ("complete" / "max-iter" / "verbatim-fail" / "spawn-err" / "install-err" / "arbitrate-err") requires the catch block

The error paths (L192-199) also need audit emit for non-complete outcomes. The audit module must be called from BOTH the success path AND the catch/error return paths.

**Integration pattern (verified against engine.ts L187-199):**

```typescript
// After completePhase on success (L189):
await completePhase({ phaseId, sessionId: capturedSessionId, status: 'complete' })
await emitAuditRecord(task, decision, matched, {
  outcome: 'complete',
  sessionId: capturedSessionId,
  iterCount: /* from ralphLoopWrap result */,
  routeLayer: 'L1-keyword',
})
return { ok: true, result, matchedRule: matched }

// In catch: emit with outcome='max-iter' / 'verbatim-fail' / 'spawn-err'
```

### 2.3 JSONL Append Strategy

**`appendFileSync` vs stream writer:**

For this use case (`appendFileSync` wins):
- Single-process Node.js tool (not a server with concurrent connections)
- `appendFileSync(path, line + '\n')` is atomic at the OS level for small writes on both Windows and Unix (write syscall for small buffers is not interrupted by other same-process calls)
- The concern about concurrent writes is only relevant when TWO simultaneous `harnessed` processes run in the same project directory — this is the "single-maintainer dogfood" phase, real-world concurrent harnessed runs are near-zero
- Sister pattern: `src/checkpoint/state.ts` uses `writeFile` (async) for checkpoint writes; audit.log can use sync append for simplicity (no need to await)
- Node.js `fs.appendFileSync` opens with `O_APPEND | O_WRONLY` — the OS guarantees atomic append for writes under PIPE_BUF (~4KB on Linux; audit records are ~200-400 bytes each)

**Windows behavior:** `O_APPEND` on Windows NTFS is also atomic for small writes via `SetFilePointer + WriteFile` at the NTFS level. The same single-maintainer dogfood rationale applies.

[ASSUMED: OS-level atomicity claims based on POSIX spec knowledge; not verified via Windows NTFS docs in this session]

**Recommendation:** `appendFileSync('.harnessed/audit.log', JSON.stringify(record) + '\n')`

**Directory creation:** `.harnessed/` already created by `src/checkpoint/state.ts` `mkdir({ recursive: true })` at first checkpoint write. audit.ts should include its own `mkdirSync` guard for the case where no checkpoint has been created yet (audit.log may be first `.harnessed/` write).

### 2.6 `route_layer` and `outcome` Derivation

**`route_layer`:** determined at the arbitrate step:
- `"L1-keyword"` — when `arbitrate(rules, taskCtx)` returns a matched rule (line 113)
- `"L2-semantic-stub"` — when L1 returns null + semanticMatch is called (v0.1 always passes through; currently always L3)
- `"L3-fallback"` — when fallbackSupervisor is invoked

In the current codebase (v0.1 semantic stub always null), the only real paths are L1 (matched) or L3 (no match + fallbackSupervisor). The route_layer determination must be passed from engine.ts to the audit hook.

**`outcome`:** mapped from EngineResult:
- `"complete"` — `ok: true` path (L190)
- `"max-iter"` — `MaxIterationsExceededError` (L193)
- `"verbatim-fail"` — `VerbatimCompleteFailError` (L196)
- `"spawn-err"` — generic spawn error (L198)
- `"install-err"` — install/factory error paths (L143, L155)
- `"arbitrate-err"` — `InvalidDecisionError` or no rule + no fallback (L97, L129)

**`iter_count`:** `ralphLoopWrap` returns a string (the result text), not an iteration count. The iter_count must be extracted from inside ralphLoopWrap OR passed out via a side-channel. Current `ralphLoopWrap` signature returns `Promise<string>`. To avoid modifying ralphLoop.ts, iter_count can be `null` in Phase 4.3 (YAGNI — defer actual iter tracking to when audit log consumer is built). This avoids a signature change to a core module. ## § 8 Risk Surface Scan

### R-1: audit log concurrent write race

**Risk:** Two simultaneous `harnessed` processes write to `.harnessed/audit.log` concurrently → interleaved JSONL lines or truncated records.

**Probability:** LOW. Single-maintainer dogfood phase; concurrent harnessed invocations are rare.

**Mitigation:** `appendFileSync` with OS-level atomic append for writes < PIPE_BUF (~4KB). Individual JSONL records are ~200-400 bytes — well within atomic range. Per sister #BU disposition: concurrent state write protection deferred to Phase 4.4+/v0.5 for the separate `state.ts` issue; same rationale applies to audit.log. [ASSUMED: OS atomic append guarantee; not formally verified this session]

**ADR 0018 should document:** "No explicit lock mechanism; atomic append via O_APPEND sufficient for single-maintainer phase; multi-process concurrent access deferred to v0.5+ if demand signal arrives."

### R-2: ADR backfill semantic gap with A7 守恒

**Risk:** ADR 0019/0020 backfill content inadvertently references or overlaps with existing ADR 0001-0017 main bodies, triggering confusion. OR: executor edits an existing ADR main body by mistake → A7 CI gate fails on next push.

**Probability:** LOW. The A7 gate is a hard CI gate that catches any diff on numbered ADR files. New files (0018-0020) are not covered by the A7 gate UNTIL `adr-0018-accepted` is pushed.

**Mitigation:** 
- Executor must ADD new files only; verify `git diff docs/adr/000[1-9]-*.md docs/adr/001[0-7]-*.md` shows 0 diff before tagging
- ADR README.md is NOT covered by A7 (not a numbered file) — updates safe
- CI will catch any accidental edit immediately on next push

### R-3: STATE.md ≤175L unreachable (sister #BA 2-iter defer risk)

**Risk:** Post-D2 iter 4 trim, STATE.md ends up in 166-175L range → DEFER #BA again (3rd consecutive defer) → SIZE_LIMIT=175 flip never happens.

**Probability:** LOW-MEDIUM. Current STATE.md = 156L. Phase 4.2 narrative estimate is ~15-25L. Post-trim estimate ~131-141L — safely below 165L threshold.

**Contingency:** If post-trim is 166-175L, accept the 3rd defer (per decision tree in deferred-items.md). SIZE_LIMIT=200 round 1 is still enforced (ENFORCE=true since Phase 4.1 W0.1). The gate is not broken; it's just not tightened yet.

**If >175L (BLOCKED scenario):** This would require ~19L growth since Phase 4.2 ship with no corresponding D2 trim — very unlikely unless the W0.1 trim task fails. Escalate and do an emergency trim.

### R-4: 1.5-2 day estimate underestimate risk

**Risk:** Phase 4.3 scope (NEW src/ module + 3 ADRs + CHANGELOG + milestone close) exceeds 2-day estimate, creating timeline pressure.

**Probability:** MEDIUM. This is significantly more complex than Phases 4.1+4.2 (~4h each). The ADR authoring component is the largest unknown — each of the 3 ADRs may require 1-2h of careful content writing.

**Mitigation strategies:**
