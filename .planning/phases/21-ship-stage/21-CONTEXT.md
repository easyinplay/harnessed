## Goal

Give harnessed a first-class ship/release stage after `verify`, closing the gap vs comet (archive), Trellis (finish-work), and Claude-Harness (`/harness-release`, "PR ready ‚âÝ release ready"). harnessed has the pieces (gstack `/ship`, checkpoint evidence, `publish.yml`) but no stage that orchestrates release-preflight + evidence + PR. The machine core is a `harnessed release-preflight` gate; the user-facing stage is a `/ship` workflow master.

## Motivating finding (verified)

- `CHANGELOG.md` has an **empty `## [Unreleased]` section** despite 8 shipped phases (13‚Äì20). This is exactly the "release evidence rebuilt by memory" failure ‚Äî a release-preflight that fails on an empty Unreleased catches it. (`## [4.4.0] - 2026-06-11` is the last entry.)

## Infrastructure (verified)

- `publish.yml`: triggers on **tag push** + `workflow_dispatch` ‚Üí `npm publish` + GitHub release. So the actual publish is CI-driven; the ship stage stops at **tag-ready** (CHANGELOG + version + tag), CI does the rest.
- `gates.ts`: `harnessed gates <master>` already evaluates workflow gates (JSON plan); ship can reuse the gate framework.
- Workflow masters live at `workflows/<stage>/auto/{SKILL.md, workflow.yaml}` (the `auto` subdir IS the master); subs at `workflows/<stage>/<sub>/{SKILL.md, workflow.yaml}`. Master `workflow.yaml`: `schema_version: harnessed.workflow.v3`, `workflow: <name>`, `disciplines_applied: [6 default]`, `tools_available: [...]`, `delegates_to: [...]` (serial needs explicit `order`; parallel doesn't ‚Äî K9 invariant). Sub `workflow.yaml` adds `phases: [...]`. Validated by `check-workflow-schema`. Routing via `workflows/judgments/stage-routing.yaml`; capabilities via `workflows/capabilities.yaml`. Slash commands auto-generated by `generateCommands.ts`. No existing ship/release command or workflow.
- CHANGELOG is Keep-a-Changelog; version in `package.json`.

## Locked decisions

- **D1 ‚Äî two layers** (chosen): (Wave 1) a CLI `harnessed release-preflight` machine gate; (Wave 2) a `/ship` workflow master (5th stage) that orchestrates preflight + evidence and delegates PR/deploy to gstack `/ship`.
- **D2 ‚Äî deploy boundary = tag-ready** (chosen): ship produces/validates CHANGELOG + version + tag-readiness; actual `npm publish` + GitHub release stay in `publish.yml` CI (tag-push triggered). ship NEVER pushes/publishes (matches comet/Trellis/Harness ‚Äî they ready + evidence, they don't deploy). No remote action from ship.
- **D3 ‚Äî release-preflight checks** (Wave 1, the machine core, read-only): CHANGELOG `[Unreleased]` non-empty; `package.json` version present; git working tree clean (no uncommitted); a git tag `v<version>` does NOT already exist; (advisory) tests ‚Äî preflight does NOT run the suite (slow), it notes "run `pnpm test`" or trusts CI. Each ‚Üí `{name, status: pass|warn|fail, message, fix}`. Exit 1 if any fail.
- **D4 ‚Äî lean `/ship` master** (Wave 2): `workflows/ship/auto/` master delegating to ONE sub `ship/preflight` (wraps `harnessed release-preflight`) + prose that delegates PR/deploy to gstack `/ship` and points at `publish.yml` for publish. Do NOT author full evidence/pr/changelog subs as separate workflow dirs (composition thesis ‚Äî delegate to gstack, don't reimplement). Must pass `check-workflow-schema` + load in the master orchestrator + auto-generate the `/ship` slash command.
- **D5 ‚Äî git read-only in preflight**: preflight reads git state (`git status --porcelain`, `git tag -l`) via an injectable runner; never mutates git.
- **D6 ‚Äî TDD red-first** for the CLI release-preflight (Wave 1). Wave 2 is composition-skill authoring validated by `check-workflow-schema` + a load smoke.

## Scope

**Wave 1 ‚Äî CLI release-preflight (TDD):**
- `src/cli/lib/release-preflight.ts`: `collectPreflight(cwd, deps): PreflightCheck[]` ‚Äî read-only checks above; `deps` injects git/fs reads for testing.
- `src/cli/release-preflight.ts`: `registerReleasePreflight` ‚Äî `harnessed release-preflight`, prints checks, exit 1 on any fail.
- `src/cli.ts`: register (26th).

**Wave 2 ‚Äî `/ship` master (authoring + schema):**

- `workflows/judgments/stage-routing.yaml`: ship trigger(s).
- `workflows/capabilities.yaml`: ship-related capability entry if needed.
- Docs: README 4-stage ‚Üí 5-stage note (minimal), the hero loop already ends at Retro ‚Äî add Ship between Verify and Retro or note it.

## Out of scope (do NOT touch)

- Actual `npm publish` / GitHub release / git push/tag (stays in `publish.yml` CI + explicit user approval ‚Äî D2).
- Full evidence/pr/changelog sub-workflows (D4 ‚Äî lean master, delegate to gstack `/ship`).
- Reimplementing gstack `/ship` or `/land-and-deploy` (delegate).
- The 4.5.0 release itself (that's the NEXT step after this phase, run through the new preflight).

## Invariants

- ship is READ-ONLY w.r.t. remote + git mutation: no push, no publish, no tag creation (preflight only checks tag absence).
- Deterministic preflight (injectable git/fs); fail-soft where a check can't run.
- Workflow additions pass `check-workflow-schema` + load in the orchestrator + generate the `/ship` command.
- KARPATHY-minimal; TDD red‚Üígreen for Wave 1; full gate green vs the post-Phase-20 baseline (1302); Windows CI green; biome preempt; NEVER `git add -A`; NEVER push without approval.
