## Implementation Decisions

### D-01: gstack 命令前缀探测策略 = PROBE (doctor 运行时探三选一 + fallback 用户选)

**Selected**: PROBE — doctor 运行时 spawn `which`/`command -v` 探三选一 + fallback prompt

**Locked rationale**: R7.4 验收"用户三种前缀场景任一都跑通"明确指向 runtime 自动检测 (而非用户手填 CONFIG OR 强制 INTERACTIVE)。PROBE zero-friction (用户无需手动配); fallback prompt 兜底 0/2 响应 ambiguity case; INTERACTIVE first-run 在 CI/headless env 跑不动 (sister Phase 2.4 doctor `--json` flag CI-friendly 经验). PROBE 与 R02 § 1 "doctor 探测" 语义完全对齐.

**Implementation hint**:
- `src/cli/doctor.ts` 加第 6 check (现 5 check + 1 = 6) — `checkGstackPrefix()` ~30L:
  - `spawnSync('which', ['gstack-office-hours'])` (Linux/macOS/Git-Bash) OR `spawnSync('where', ['gstack-office-hours.exe'])` (Win) 探 `gstack-` 前缀
  - 同样探 `office-hours` (`--no-prefix` 默认)
  - 三种状态: `gstack-` only / `office-hours` only / both OR neither
  - both / neither → fallback `console.log` 提示用户手填 + exit 1 (Karpathy fail-loud); 单一命中 → 自动写 `.harnessed/config.json`
- `.harnessed/config.json` schema 加 `gstack_prefix: 'gstack-' | '' | 'custom-<x>'` field (TypeBox + branchOnSchemaVersion); Wave A R2 决是否 9th schemaVersion surface OR 复用 existing
- `--json` flag 输出 prefix 检测结果 (CI-friendly sister doctor.ts pattern)
- Karpathy hard limit `doctor.ts` 215L (B-03 5% tolerance 已用 Phase 2.4); 本 phase 加 ~30L → 215→245L 超 → 需 split helper `src/cli/lib/probe-gstack.ts` ≤50L (sister Phase 2.4 W3 engineHook.ts PRIMARY pattern)

**decision_source**: 用户首轮决策 PROBE (Recommended option)

### D-02: workflow `invokes` 变量插值机制 = JINJA-like `{{ prefix }}` 模板替换

**Selected**: JINJA-like `{{ prefix }}` 模板替换 (zero-dep str.replace)

**Locked rationale**: zero-dep (沿袭 Karpathy YAGNI + sister Phase 2.4 dashboard template parse 模式); cross-OS (str.replace 行为一致, 不依赖 envsubst Win 行为); 模板 syntax `{{ var }}` 用户直观可读 (vs `${VAR}` 需 shell context 上下文); SCHEMA 模板化字段 over-engineering (Karpathy YAGNI: workflow yaml 字段单类型即可). 与 R7.4 "workflow `invokes` 字段变量插值" 语义对齐.

**Implementation hint**:
- `src/workflow/interpolate.ts` NEW ~30L — `interpolate(template: string, vars: Record<string, string>): string`:
  - regex `/\{\{\s*(\w+)\s*\}\}/g` 找所有 `{{ var }}` placeholder
  - 替换 vars[name]; 未定义 var → throw `InterpolationError` (Karpathy fail-loud)
  - 不引第三方 (mustache / handlebars dep)
- workflow.yaml loader (`src/workflow/loadPhases.ts` 既存) extend 加 `interpolateInvokes(yamlContent, vars)` step
- vars 来源: `.harnessed/config.json` `gstack_prefix` + Phase 3.4 可能加 model_tier 等
- Karpathy hard limit `interpolate.ts` ≤30L

**decision_source**: 用户首轮决策 JINJA (Recommended option)

### D-04: CEO veto halt_workflow 触发 mechanism = PUSH (file-based)

**Selected**: PUSH — gstack veto 写 `.harnessed/governance.json` → plan-feature workflow next-phase 读

**Locked rationale**: zero coordination (无 socket/IPC dep); 沿袭 Phase 3.1 `.harnessed/current-workflow.json` state machine pattern (file-based state); POLL 是 sister anti-pattern (Phase 2.4 SSE polling 已 root-cause defer 改 EventSource); EVENT signal cross-OS 不可靠 (Win signal 与 Linux 不同). PUSH "下一 phase 转换前 poll 一次" 是 lazy read, 不是定时 polling — overhead 仅 5 phase × 1 read.

**Implementation hint**:
- `.harnessed/governance.json` schema (`harnessed.governance.v1` 9th schemaVersion surface OR 复用 config.v1):
  ```ts
  { schemaVersion: 'harnessed.governance.v1', status: 'active' | 'vetoed', reason?: string, vetoed_at?: string, vetoed_by?: string }
  ```
- `src/workflow/governance.ts` `readGovernance()` + `isVetoed()` (D-03 hint)
- `src/workflow/run.ts` next-phase 转换前 1 行 `if (isVetoed()) { engineHook({status:'paused'}); return }` — 复用 Phase 3.1 engineHook (D-04 WIRE-IN consumer 二代)
- gstack 写 governance.json 不在本 phase scope (gstack 自身职责; harnessed 只 reader)
- 用户 resume: `harnessed resume` → 读 checkpoint paused state → 提示用户 (D-03 RELOAD); 若 governance 仍 vetoed 则 governance 解除 后再 resume (Phase 3.3+ workflow 自然 retry)

**decision_source**: 用户首轮决策 PUSH (Recommended option)
