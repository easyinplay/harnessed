## § 1 — cc-plugin-marketplace

**Targets**: anthropics/* plugins, ui-ux-pro-max-系, frontend-design-系 plugins distributed via CC marketplace.

### Current CLI / API (2026-05)

**MAJOR CHANGE since 2026-05-11 original research** — Claude Code now ships first-class
**non-interactive `claude plugin` CLI subcommands** (added in response to issue #12999; previously
plugin management was interactive `/plugin` slash-command only). This unblocks headless install.

Two-step flow:

```bash
# Step 1 — register the marketplace (idempotent: re-add auto-updates)
claude plugin marketplace add <url-or-owner/repo>
#   e.g. claude plugin marketplace add anthropics/claude-code-marketplace

# Step 2 — install a plugin from a registered marketplace
claude plugin install <plugin>@<marketplace-name> [--scope user|project|local]
#   e.g. claude plugin install plugin-dev@claude-code-marketplace --scope project
```

Supporting subcommands: `claude plugin marketplace list|update|remove`,
`claude plugin list [--json] [--available]`, `claude plugin uninstall <plugin> [--scope] [-y] [--keep-data] [--prune]`,
`claude plugin enable|disable|update`, `claude plugin validate`.

**Marketplace manifest format**: `.claude-plugin/marketplace.json` at marketplace repo root. Each
plugin entry has `source` (path/github/url/git-subdir), optional `version` (semver — if omitted,
falls back to git commit SHA, so every commit = new version). `plugin.json` `version` wins over
marketplace entry. Plugins are **copied to a local cache** `~/.claude/plugins/cache` (not used
in-place) for security/verification.

**Files modified**: marketplace registration + `--scope user` → `~/.claude.json`;
`--scope project` → `<project>/.claude/settings.json` `enabledPlugins` key; `--scope local` →
gitignored local settings. Plugin data dir: `~/.claude/plugins/data/{id}/`.

### Headless support

**YES — fully headless.** `claude plugin install` has **no TTY prompt** — it is non-interactive by
design (only `uninstall`/`prune` have a confirmation needing `-y, --yes` when stdin/stdout is not a
TTY). `claude plugin install` itself takes only `-s, --scope` and `-h`. So the harnessed installer
can spawn it directly with no flag gymnastics.

### Failure modes

- Marketplace `add` of an already-registered marketplace → CC errors (not idempotent on first add;
  but `marketplace add` of an existing one auto-updates per community report — **verify exact
  behavior during impl**, treat as MEDIUM).
- `Plugin directory not found at path: ...` — marketplace.json `source` points to missing dir.
- `Plugin <x> has conflicting manifests` — both plugin.json and marketplace entry specify components.
- `claude plugin validate` failures — invalid plugin.json / frontmatter / hooks.json.
- Network failure reaching marketplace git source.
- `claude plugin prune` requires CC v2.1.121+; manifest `experimental.*` schema churn (v2.1.140+
  flags ignored folders in `/doctor`).

### ADR 0004 6-contract fit

- **C1 dry-run**: no native `--dry-run` on `claude plugin install`. Simulate diff textually like
  mcpStdioAdd does (reconstruct the `enabledPlugins` entry that will land in settings.json).
- **C2 diff**: show unified diff of `~/.claude.json` or `.claude/settings.json` gaining the entry.
- **C3 rollback**: backup `~/.claude.json` / `.claude/settings.json` before; rollback restores.
  Note: plugin cache dir `~/.claude/plugins/cache` also written — uninstall via `claude plugin
  uninstall` is the clean path (4-step fallback uninstall is a Phase 2.4 item).
- **C4 confirm level**: **L3** (modifies `~/.claude.json` / settings.json — matches index.ts
  `levelOf` seed). `--scope project` is the safer default; only `--scope user` touches global.
- **C5 CLI-only**: N/A (C5 is MCP-specific) — but the same *spirit* applies: use `claude plugin`
  CLI, never hand-edit `~/.claude.json` `enabledPlugins`.
- **C6 no silent failure**: verify with `claude plugin list --json` exit code + grep (mirror
  mcpStdioAdd's `claude mcp list | grep -q` pattern — exit code is the stable contract, not stdout
  text).

**Confidence: HIGH** — official docs (code.claude.com/docs/en/plugins-reference) confirm exact
subcommands, flags, scopes, files. The one MEDIUM gap: exact `marketplace add` idempotency edge.

## § 2 — git-clone-with-setup

**Reference**: `manifests/skill-packs/ui-ux-pro-max.yaml` — already a real git-clone-with-setup
manifest. Phase 1.3 实测 PATH_A+B 双 OK (D1.3-5 path B 主推 — avoids vercel-labs/skills issue #373
non-main-branch update break). Manifest `cmd` does: `rm -rf cache && git clone --depth 1 --branch
v4-next <repo> cache && cp -R cache/.codex/skills/ui-ux-pro-max ~/.claude/skills/ && rm -rf cache`.
`git_ref: e89d70e4bcd0...` (full SHA pinned).

### Current CLI / API (2026-05)

**CHANGE since original research**: Git 2.49.0 (March 2025) added `--revision` for true
single-command commit-pinned shallow clone:

```bash
git clone --revision <full-40-char-SHA> --depth 1 <https-url> <dest>
```

Canonical safe flow (works on older Git too — the manifest already encodes the `--branch` variant):

1. `rm -rf <cache-dir>` (clean slate — idempotency)
2. Clone: `--revision <SHA> --depth 1` (Git 2.49+) **OR** `--depth 1 --branch <branch>` then verify
   (older Git — what ui-ux-pro-max.yaml uses since the skill lives on `v4-next` branch).
3. **Verify HEAD**: `git rev-parse HEAD` matches the pinned `git_ref` SHA before running anything.
4. Run setup / copy step (`cp -R ... ~/.claude/skills/`).
5. `rm -rf <cache-dir>` cleanup.

### Headless support

**YES — fully headless.** Plain git + shell. No interactive prompts (HTTPS public repo = no auth
prompt). The manifest `cmd` string is a single shell pipeline. Existing `lib/spawn.ts` +
`checkCmdString` security gate already screens it (ADR 0001 B1 — `${...}` / `$(...)` / `eval`
forbidden; `git_ref` HEAD/main/master forbidden by preflight `git_ref` keyword).

### Failure modes

- `git_ref` is a branch tip / HEAD / main / master → **preflight rejects** (`git_ref` keyword, exit
  1) — already enforced. ui-ux-pro-max pins full SHA ✅.
- Shallow clone of unadvertised SHA → `Server does not allow request for unadvertised object`.
  Mitigation: the `--branch <branch> --depth 1` variant (already used) sidesteps this; or
  incremental-depth fetch fallback.
- Setup script is **untrusted code** — runs with user privileges. harnessed mitigations: full-SHA
  pin (immutable), HTTPS authenticated transport, `git rev-parse HEAD` verify before exec, and the
  manifest `cmd` is print-in-full at confirm time (C2 diff preview shows it).
- Disk: `cp -R` into `~/.claude/skills/` can collide with existing dir → idempotent_check
  (`test -f ~/.claude/skills/<name>/SKILL.md`) guards; `--force` (Phase 2.1) to override.
- Cross-OS: `rm -rf` / `cp -R` are Unix — **Windows native needs Git Bash** (already a known
  doctor check, Phase 2.4). The `cmd` as-written is Git-Bash-dependent on win32; manifest declares
  `platforms: [linux, darwin, win32]` so win32 path MUST run under Git Bash. Flag this.

### ADR 0004 6-contract fit

- **C1 dry-run**: print the full `cmd` pipeline + target paths; no write until apply.
- **C2 diff**: `+++ NEW: ~/.claude/skills/<name>/` directory tree summary. The `cmd` string itself
  is shown verbatim (it's effectively the "diff" of what executes).
- **C3 rollback**: backup is mostly **pure-create** (skills dir didn't exist) → `metadata.json`
  `"backup": ""` ENOENT sentinel → rollback = `rm -rf` the created dir. EOL preservation N/A
  (skill files copied as-is).
- **C4 confirm level**: **L2** default (writes `~/.claude/skills/` — matches index.ts seed).
  **L4 if** the repo's setup.sh installs a binary to PATH — manifest must declare this; absent a
  setup.sh that touches PATH, ui-ux-pro-max is pure L2 (copy-only).
- **C5**: N/A (not MCP).
- **C6 no silent failure**: `verify.cmd` (`test -f ~/.claude/skills/<name>/SKILL.md`) exit code is
  the gate — already in the manifest. Clone fail / copy fail → STOP, no partial.

**Confidence: HIGH** — ui-ux-pro-max.yaml is a real working manifest (Phase 1.3 实测 PATH_A+B OK);
git mechanics are stable and verified against git-scm.com. Main caveat: win32 Git Bash dependency.

## § 3 — npx-skill-installer

**Targets**: Claude Code skills installed via an npx-runnable CLI (anthropics/skills, vercel-labs
skills, etc.).

### Current CLI / API (2026-05)

The ecosystem standard is **`npx skills`** (vercel-labs/skills) — a GitHub-registry-backed skill
package manager. There is **no single official Anthropic `npx` skill installer**; vercel-labs/skills
is the de-facto tool, with `openskills`, `add-skill`, `agent-skills-cli` as alternatives.

```bash
# Add a skill from a GitHub repo, non-interactive, targeting Claude Code
npx skills add <owner/repo> --agent claude-code -y
#   e.g. npx skills add anthropics/skills --skill claude-api --agent claude-code -y

# Specific skill from a multi-skill repo
npx skills add https://github.com/vercel-labs/skills --skill find-skills -a claude-code -y
```

Source formats: `owner/repo` shorthand, full GitHub URL, direct path-to-skill-in-repo, GitLab URL,
any git URL, local path. Flags: `--skill <name>` (specific), `--all` (all skills in repo),
`--list` (enumerate without installing), `-g/--global` (user-level), `-a/--agent <agent>`
(repeatable), `-y` (non-interactive CI-friendly).

**Files modified**: For Claude Code, skills land in `.claude/skills/` (project) or
`~/.claude/skills/` (with `-g`).

**KNOWN GOTCHA — directory convention conflict**: `npx skills` (Agent Protocol ecosystem) may
install to `~/.agents/skills/` as a tool-agnostic home, while Claude Code reads `~/.claude/skills/`.
Symptom: "skills not found after npx install". The `--agent claude-code` flag is supposed to target
the right dir — **verify during impl that `--agent claude-code` writes directly to
`~/.claude/skills/`** and does not require a SessionStart sync hook bridge. If it doesn't,
git-clone-with-setup (§ 2) is the safer path for these skills (copy directly to `~/.claude/skills/`).

### Headless support

**YES** — `-y` flag is explicitly "non-interactive CI/CD-friendly". `npx` itself needs `--yes` to
skip the install-confirmation (the npmCli installer already handles `npx --yes` pattern).

### Failure modes

- **Directory convention mismatch** (above) — biggest risk; skill installs "successfully" but CC
  doesn't see it. **No-silent-failure C6 violation risk** — verify must check the *actual*
  `~/.claude/skills/<name>/SKILL.md` path, not just the npx exit code.
- npx fetches latest tool version each run — **non-reproducible** unless the manifest pins the
  installer tool version (`npx skills@<version>`). ADR 0007 `install_type: npx` field applies here.
- Network / npm registry failure.
- Multi-skill repo with `--skill` name containing spaces → must be quoted.
- Tool fragmentation: vercel-labs/skills vs openskills vs add-skill — manifest must pin which tool.

### ADR 0004 6-contract fit

- **C1 dry-run**: use `npx skills add ... --list` to enumerate what *would* install without
  writing; or print the planned `~/.claude/skills/<name>/` target.
- **C2 diff**: `+++ NEW: ~/.claude/skills/<name>/`.
- **C3 rollback**: pure-create backup sentinel (same as § 2) → rollback = `rm -rf` skill dir.
- **C4 confirm level**: **L2** (writes `~/.claude/skills/` — matches index.ts seed).
- **C5**: N/A.
- **C6 no silent failure**: **critical** — verify by `test -f ~/.claude/skills/<name>/SKILL.md`
  exit code (NOT npx exit code), because of the directory-convention gotcha. If verify fails after
  npx reports success → report as `verify-failed` with `suggest:` pointing at the
  `~/.agents/` vs `~/.claude/` bridging issue.

**Confidence: MEDIUM** — `npx skills` (vercel-labs) CLI flags verified via WebSearch, but: (a) no
single official Anthropic installer — tool choice is a judgment call; (b) the `~/.agents/` vs
`~/.claude/` directory-convention conflict is a real documented gotcha that could make this method
unreliable. **Recommendation**: see § 5 — consider whether git-clone-with-setup (§ 2) covers the
same skills more reliably, and keep npx-skill-installer minimal / pin one tool.

## § 4 — mcp-http-add

**Reference**: `src/installers/mcpStdioAdd.ts` — the working sibling. v0.1.0 stdio installer
**hardcodes `--scope project`** to dodge CC v2.1.122 user-scope bug (#54803: user-scope MCP config
silently lands in `~/.claude.json` but CC fails to read it back).

### Current CLI / API (2026-05)

```bash
claude mcp add --transport http --scope project <name> <url> [--header "Key: Value"]
#   e.g. claude mcp add --transport http --scope project plane https://mcp.plane.so/http/mcp
```

**Flag ordering is strict**: all options (`--transport`, `--scope`, `--header`, `--env`) MUST come
*before* `<name>`. For HTTP transport you give a `<url>` instead of a `-- <command>`. `--header` is
repeatable, supports `${ENV_VAR}` interpolation for auth tokens.

**HTTP vs SSE**: HTTP (streamable HTTP) is the current standard; SSE is legacy/migrating. Use
`--transport http` unless a server specifically requires SSE.

**Files modified**: `--scope project` → `<project>/.mcp.json` `mcpServers` map (same file
mcpStdioAdd writes). `--scope user`/`local` → `~/.claude.json`.

### Does the user-scope bug apply to HTTP?

**Yes — same mitigation applies.** Bug #54803 is about CC failing to *read back* user-scope MCP
config from `~/.claude.json` — it is **transport-agnostic** (the read-back path is shared by stdio
and HTTP entries). Therefore mcp-http-add should **hardcode `--scope project`** identically to
mcpStdioAdd: authoritatively reconstruct the args array, treat manifest `cmd` as audit-trail only,
ignore any manifest attempt to override scope. **Confidence on "same bug applies": MEDIUM** — the
bug is documented for MCP generally and the read-back path is shared, but I did not find an HTTP-
transport-specific reproduction. Safe default: hardcode `--scope project` (it's the correct scope
per ADR 0004 C5 regardless).

### Headless support

**YES** — `claude mcp add` is non-interactive (same as the stdio path mcpStdioAdd already spawns).
**Caveat**: HTTP servers requiring **OAuth** trigger an interactive browser auth flow on first
*use* (not on `add`). `add` itself just writes config. For token-auth servers, `--header
"Authorization: Bearer ${TOKEN}"` is fully headless. OAuth servers → `add` succeeds headless but
`claude mcp get <name>` shows OAuth-pending state. Flag OAuth servers as needing manual auth.

### Failure modes

- Flag-ordering mistake (options after `<name>`) → CLI parse error. The installer constructs the
  args array authoritatively (like mcpStdioAdd) so this is controlled.
- User-scope read-back bug #54803 (mitigated by hardcoded `--scope project`).
- `claude mcp add --dry-run` flag is **not documented in v2.1** — mcpStdioAdd already avoids it and
  simulates the `.mcp.json` entry textually; mcp-http-add must do the same.
- URL unreachable at `add` time — CC may or may not validate connectivity on add; verify step
  catches it.
- OAuth servers: `add` succeeds but server unusable until interactive auth.
- Header secrets: `${ENV_VAR}` interpolation — ensure the env var is set; never bake literal
  tokens into the manifest (security gate B1 — though `${...}` is normally *forbidden* by
  checkCmdString; **header values are an exception that needs care** — the `--header` value is
  data, not a shell expansion, but checkCmdString may false-positive on `${...}`. **Impl note**:
  may need a checkCmdString carve-out for `--header` arg values, OR require env vars be resolved
  before constructing args. Flag this for plan-phase.).

### ADR 0004 6-contract fit

- **C1 dry-run**: simulate `.mcp.json` entry textually (copy mcpStdioAdd's approach — no native
  `--dry-run`).
- **C2 diff**: `.mcp.json` gains `{name: {type: "http", url: "...", headers: {...}}}` entry.
- **C3 rollback**: backup `<project>/.mcp.json` before; rollback restores. Same as mcpStdioAdd.
- **C4 confirm level**: **L3** (writes `.mcp.json` — matches index.ts seed).
- **C5 CLI-only**: **enforced** — use `claude mcp add` CLI, hardcode `--scope project`, never
  hand-edit `.mcp.json`. Manifest cannot override scope. **This is the primary contract for this
  method** and mcpStdioAdd is the 1:1 reference pattern.
- **C6 no silent failure**: verify via `claude mcp list | grep -q <name>` exit code (NOT stdout
  parse) — copy mcpStdioAdd verbatim. For OAuth servers, `claude mcp get <name>` to surface
  pending-auth state rather than reporting false success.

**Confidence: HIGH** on CLI syntax + scope handling (verified official docs + mcpStdioAdd is the
proven sibling). **MEDIUM** on "user-scope bug applies identically to HTTP" (transport-agnostic
read-back path strongly implies yes; no HTTP-specific repro found — but hardcoding `--scope
project` is correct regardless).

## § 5 — v0.2.0 Phase 2.1 Implementation Recommendations

### Implementation order (lowest-risk first)

1. **mcp-http-add FIRST** — mcpStdioAdd is a near-1:1 reference. Copy the file structure:
   authoritative args array, `checkCmdString` per-arg re-screen, hardcoded `--scope project`,
   textual `.mcp.json` diff simulation, `claude mcp list | grep -q` verify. Only deltas:
   `--transport http` + `<url>` instead of `-- npx ...`, and `--header` handling. **Highest
   confidence, fastest win, proven pattern.**

2. **git-clone-with-setup SECOND** — ui-ux-pro-max.yaml is a real working manifest, Phase 1.3
   实测 PATH_A+B OK. The `cmd` pipeline already passes the B1 security gate. Installer mostly:
   preflight (`git_ref` SHA check already enforced) → diff (print `cmd` + target) → backup
   (pure-create sentinel) → spawn `cmd` via lib/spawn.ts → verify (`test -f .../SKILL.md`).
   **L2 default, L4 only if setup.sh touches PATH.** Watch: win32 Git Bash dependency.

3. **cc-plugin-marketplace THIRD** — now genuinely unblocked (the big refresh finding: official
   `claude plugin` CLI subcommands exist, fully headless). Two-step: `marketplace add` then
   `plugin install <plugin>@<marketplace> --scope project`. L3. Verify via `claude plugin list
   --json`. Slightly more moving parts (marketplace registration is stateful) than the above two.

4. **npx-skill-installer LAST** — **MEDIUM confidence, most uncertainty.** The `~/.agents/` vs
   `~/.claude/skills/` directory-convention conflict is a documented real gotcha. **Decision point
   for plan-phase**: either (a) implement minimally, pin ONE tool (`npx skills@<ver>` from
   vercel-labs), and make verify check the *actual* `~/.claude/skills/<name>/SKILL.md` path; or
   (b) **keep partially deferred** — if Phase 2.3's design/content/testing extension candidates
   can all be installed via git-clone-with-setup (§ 2) instead, npx-skill-installer may not be on
   the critical path. ui-ux-pro-max already uses git-clone, not npx.

### Should any method stay deferred?

- **None hard-blocked.** The 2026-05-11 → 2026-05-15 refresh actually *improved* the situation:
  `claude plugin` CLI subcommands are now first-class and headless (was the biggest unknown).
- **npx-skill-installer is the soft-defer candidate** — not because upstream isn't ready, but
  because (a) no single official tool, and (b) the directory-convention gotcha makes reliable
  verify hard. Recommend: implement a thin version pinning vercel-labs/skills, OR document it as
  "best-effort, prefer git-clone-with-setup" and let Phase 2.3 extension-category work decide if
  it's actually needed. **Do not let it block the other 3.**

### Cross-cutting impl notes

- **All 4** must declare their max Level via `levelOf()` — index.ts seed already correct (L3 / L2 /
  L2 / L3). No change needed.
- **All 4** lack a native `--dry-run` → all simulate the diff textually (mcpStdioAdd established
  this pattern; reuse it).
- **All 4** verify via **exit code, not stdout parsing** (ASSUMPTIONS C2 — `claude mcp list | grep
  -q`, `claude plugin list --json` exit, `test -f .../SKILL.md`).
- **mcp-http-add `--header` + checkCmdString**: `${ENV_VAR}` in header values may false-positive
  on the B1 shell-escape gate. Needs a plan-phase decision: carve-out for `--header` values, or
  resolve env vars before arg construction.
- **win32**: git-clone-with-setup `cmd` pipelines (`rm -rf` / `cp -R`) require Git Bash — Phase 2.4
  doctor check covers detection, but Phase 2.1 verify on win32 CI will exercise it.
