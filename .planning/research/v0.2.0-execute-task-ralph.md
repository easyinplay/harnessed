### 1.2 Real `query()` API shape (verified ctx7 2026-05-15)

```typescript
import { query } from "@anthropic-ai/claude-agent-sdk";

for await (const message of query({
  prompt: "Review the authentication module for security issues",
  options: {
    allowedTools: ["Read", "Grep", "Glob", "Task"],   // "Task" REQUIRED for subagent invocation
    agents: {
      "code-reviewer": {
        description: "Expert code review specialist. Use for quality/security reviews.",
        prompt: "You are a code review specialist...",
        tools: ["Read", "Grep", "Glob"],
        model: "sonnet",
      },
    },
  },
})) {
  if (message.type === "result" && message.subtype === "success") {
    console.log(message.result);
  }
}
```

Key facts:
- `query()` returns an **async iterable** of messages — main-process drives the loop by `for await`.
- Subagents are passed as the **`agents` map** in `options` — keyed by name, value is `AgentDefinition`.
- **`"Task"` must be in `allowedTools`** for the main agent to spawn subagents. (harnessed factory contract § 2.3 already forbids `Agent`/recursive spawn — `Task` is the actual tool name; reconcile naming in Phase 2.2.)
- Subagent context detection: messages from inside a subagent carry `parent_tool_use_id`; the spawn itself is a `tool_use` block with `name === "Task"` and `input.subagent_type`.

### 1.3 Real `AgentDefinition` shape vs harnessed's 14-field contract

**The official SDK `AgentDefinition` type is only 4 fields** (ctx7 `typescript.md`, fetched 2026-05-15):

```typescript
type AgentDefinition = {
  description: string;        // required
  prompt: string;             // required
  tools?: string[];           // optional
  model?: 'sonnet' | 'opus' | 'haiku' | 'inherit';   // optional

```

**⚠️ TRANSPARENCY FLAG (phase 1.5 H1 lesson — "interface exists" ≠ "production-ready")**: harnessed's `AGENT-DEFINITION-FACTORY-CONTRACT.md` documents **14 fields** (`disallowedTools`, `skills`, `mcpServers`, `memory`, `maxTurns`, `background`, `effort`, `permissionMode`, `initialPrompt`, `criticalSystemReminder_EXPERIMENTAL`, ...). The **public ctx7-indexed `AgentDefinition` type only exposes 4**. This is a real gap:

- The extra 10 fields are either (a) on `ClaudeAgentOptions` / `query()` options rather than per-`AgentDefinition`, or (b) accepted-but-undocumented, or (c) from `code.claude.com/docs` pages not mirrored in the ctx7 index.

## § 2 ralph-wiggum plugin: switch vs keep self-implemented

**Confidence: HIGH** — cross-verified official README + 4 independent 2026 articles.

### 2.1 What the official `ralph-wiggum` plugin actually is

- Official Anthropic plugin in `anthropics/claude-code/plugins/ralph-wiggum`. Install: `/plugin install ralph-wiggum@claude-plugins-official`.
- **Mechanism**: a **Stop hook** (`hooks/stop-hook.sh`) that intercepts the **interactive TUI session exit** (exit code 2 blocks Claude from stopping), re-feeds the original prompt. Loop continues until `--completion-promise` string appears OR `--max-iterations` hit.
- Commands: `/ralph-loop "<prompt>" --max-iterations <n> --completion-promise "<text>"`, `/cancel-ralph`.

### 2.2 Why harnessed should NOT switch

| Blocker | Detail |
|---------|--------|
| **Interactive-only — no headless/programmatic mode** | The README documents ZERO headless/SDK/library interface. The Stop-hook fires on **interactive session exit**. harnessed's engine is a **main-process Node.js orchestrator calling `query()`** — there is no interactive session to intercept. Architecturally incompatible. |
| **jq dependency breaks Windows** | The plugin has an **undocumented jq dependency that breaks on Windows/Git Bash** — this is *literally* R5.3's red flag. Switching to the plugin would re-import the exact bug harnessed exists to avoid. |
| **Session bleed + Docker issues** | Known issues: session bleed across terminals, Docker incompatibility. |
| **Issue #23084 "coercive language"** | Community/model-welfare debate over the Stop-hook's "do not lie to exit" prompt language. Reputational + ethical surface harnessed should not adopt wholesale. |
| **`--completion-promise` officially de-recommended** | Official README + multiple articles: exact-string-match is **unreliable**; `--max-iterations` is "your real safety net". Cannot express multi-status (`SUCCESS` vs `BLOCKED`). |

### 2.3 Note: native `/loop` is NOT relevant

Anthropic shipped a native `/loop` command — but it is **scheduled polling / cron** (`/loop 5m check deploy health`, session-scoped, 7-day auto-expiry), NOT iterate-until-done. "Ralph iterates toward completion; /loop polls on a schedule." execute-task needs iterate-until-done → `/loop` does not apply. Do not adopt.

### 2.4 Decision: KEEP self-implemented `ralphLoopWrap`

**Verdict: KEEP `src/routing/lib/ralphLoop.ts` (the ≤50L self-implemented wedge).**

Rationale:
1. harnessed's loop is **SDK-driven, not TUI-Stop-hook-driven** — the self-implemented wrap is the *correct* architecture for a main-process orchestrator, not a stopgap.
2. The plugin's jq dep would re-introduce R5.3's Windows bug — keeping self-implemented is what *delivers* R5.3 (pure-Node loop, no jq, no bash).
3. The `<promise>COMPLETE</promise>` XML-wrapper convention phase-1.5 adopted **stays as the prompt-level contract** — it is convention-compatible with the plugin (so the prompt text is portable) without taking the plugin's broken runtime.
4. **Upgrade the loop body** (see § 3 + § 4): wrap each iteration as one `query()` call, drive exit off `SDKResultMessage.subtype`, use `structured_output` for completion. This is still ≤50-80L, still self-implemented, just SDK-backed instead of dry-run-stub-backed.

phase-1.5 R6 mitigation said "切换官方 plugin 仍推 v0.2+ ADR 0010+ evaluation window". **This research closes that evaluation: do NOT switch. Self-implemented is the permanent architecture.** Record as ADR 0010.

## § 3 execute-task workflow pattern

**Confidence: HIGH** — SDK orchestration primitives verified ctx7; 5-phase pipeline maps to REQUIREMENTS R6.1.

### 3.1 The 5-phase pipeline (R6.1) mapped to SDK primitives

```
clarify (brainstorming) → code (karpathy 心法) → test (conditional TDD) → deliver (ralph-loop COMPLETE) → pause (human review)
```

Recommended orchestration shape — **one `query()` per subtask, ralph-loop wraps the deliver phase**:

```typescript
// src/routing/executeTask.ts (Phase 2.2 — main-process orchestrator)
async function executeSubtask(subtask: Subtask, decision: ArbitrateResult): Promise<SubtaskResult> {
  let sessionId: string | undefined;
  const maxIterations = decision.maxIterations ?? 20;   // R6.4 — required, never unbounded

  for (let i = 0; i < maxIterations; i++) {
    const agentDef = await agentFactory(taskCtx, decision, opts);   // existing factory
    const q = query({
      prompt: i === 0 ? buildInitialPrompt(subtask, agentDef) : buildRetryPrompt(subtask, lastResult),
      options: {
        allowedTools: ["Read", "Edit", "Write", "Grep", "Glob", "Bash", "Task"],
        agents: { [decision.primary_expert]: agentDef },
        outputFormat: { type: "json_schema", schema: COMPLETION_SCHEMA },   // § 4
        ...(sessionId ? { resume: sessionId } : {}),                        // carry context
      },
    });

    let result: SDKResultMessage | undefined;
    for await (const msg of q) {
      if (msg.type === "system" && msg.subtype === "init") sessionId = msg.session_id;
      if (msg.type === "result") result = msg;
    }

    if (!result) throw new SpawnFailError(subtask.id);          // § 5 path 3
    if (result.subtype === "error_max_turns") continue;        // retry — per-spawn turn cap hit
    if (result.subtype === "error_during_execution") { /* backoff + retry */ continue; }
    if (isComplete(result)) return { status: "COMPLETE", result };   // § 4
    // else: not complete, no error → ralph-loop retry with augmented prompt
  }
  return { status: "BLOCKED", reason: "max_iterations" };       // escalate to human → pause phase
}
```

Key patterns:
- **`resume` for retry continuity** — iteration N resumes iteration N-1's `session_id`. The subagent natively sees its prior work + the failure context. Eliminates phase-1.3 contract § 5.2's "manual augmented `task.task` re-injection" hack — `resume` does it natively. (Keep the augmented *retry prompt* for the explicit "fix X and emit completion" instruction, but context carry is free.)
- **Two iteration caps, both required** (R6.4 + AGENT-DEFINITION-FACTORY-CONTRACT § 6): outer `maxIterations` (ralph-loop, default 20) × inner per-spawn turn cap (SDK-side; phase-1.3 contract recommends 50). Worst case `20 × 50 = 1000` turns before BLOCKED.
- **`subtype` discriminated union drives exit** — `success` → check completion; `error_max_turns` → retry; `error_during_execution` → backoff+retry; `error_max_budget_usd` → BLOCKED escalate. No string matching for control flow.

### 3.2 Phase boundaries — clarify/code/test stay prompt-level

The clarify→code→test phases are **inside the subagent's prompt + skills overlay**, not separate `query()` calls. The main process does NOT orchestrate 5 separate spawns per subtask — it spawns one subagent whose `prompt` (built by `agentFactory` + `mattpocock_phases` skills overlay from decision_rules.yaml v2) embeds the 5-phase discipline. ralph-loop wraps the *whole* thing; the "deliver COMPLETE" + "pause human review" are the loop-exit + main-process-return boundary. This keeps it karpathy-simple — no orchestration explosion.

### 3.3 karpathy 心法 + mattpocock 招式 injection

- karpathy 心法 (always-on behavior rule, R6.2): injected via the factory's `prompt` prepend — phase-1.4 D1.4-14 pattern, or via `criticalSystemReminder_EXPERIMENTAL` if that field survives `.d.ts` verification (§ 1.3). CLAUDE.md merge is a *separate* Phase 2.3 concern, NOT Phase 2.2.
- mattpocock 招式 (on-demand, R6.3): `decision_rules.yaml v2 mattpocock_phases.execute.skills` → `skills_overlay` → factory translates into the subagent prompt's "available skills" section. The shipped decision_rules.yaml v2 already has this schema (phase-1.5 8 接口契约 #5).

## § 4 Completion verification best practice

**Confidence: HIGH** — `outputFormat`/`structured_output` verified in ctx7 `structured-outputs.md` guide.

### 4.1 The problem with verbatim `<promise>COMPLETE</promise>` grep

phase-1.4 Path-4 (AGENT-DEFINITION-FACTORY-CONTRACT § 5.4) flagged it: if the main agent **summarizes** the subagent's final message instead of passing it verbatim, the `<promise>` grep gets a false negative → ralph-loop retries forever. phase-1.5 upgraded raw `^COMPLETE$` → `<promise>([^<]+)</promise>` (more robust against mid-paragraph false positives) but the **summarize-risk remained** — it is still a string-match on free-text LLM output. The official ralph-wiggum docs independently confirm: `--completion-promise` exact-string-match is **"unreliable"**.

### 4.2 The better pattern: SDK structured output

The SDK now has first-class structured output (NEW since phase-1.5 baseline):

```typescript
const COMPLETION_SCHEMA = {
  type: "object",
  properties: {
    status: { type: "string", enum: ["COMPLETE", "PARTIAL", "BLOCKED"] },
    summary: { type: "string" },
    blockers: { type: "array", items: { type: "string" } },
  },
  required: ["status"],
};

// in query(): options.outputFormat = { type: "json_schema", schema: COMPLETION_SCHEMA }
// result.structured_output is then a TYPED object — no string matching
function isComplete(result: SDKResultMessage): boolean {
  return result.subtype === "success"
      && (result.structured_output as { status?: string } | undefined)?.status === "COMPLETE";
}
```

Why this is strictly better:
- **`structured_output` is a typed field on the result message** — not free text. The "main agent summarized it" risk is **structurally impossible** — the field is populated by the SDK from a schema-validated turn, not by the main agent paraphrasing.
- **Multi-status for free** — `COMPLETE` / `PARTIAL` / `BLOCKED` in one schema. The thing official ralph-wiggum docs say you *can't* do with `--completion-promise`, you get natively.
- **Built-in retry + typed failure** — `error_max_structured_output_retries` subtype tells you the subagent couldn't produce valid output (a distinct, actionable failure mode).

### 4.3 Decision: dual-signal, structured_output PRIMARY + `<promise>` grep FALLBACK

**Verdict: D2.2-3 — make `outputFormat: json_schema` → `structured_output` the PRIMARY completion signal; keep `<promise>COMPLETE</promise>` grep as FALLBACK.**

- **Primary**: every execute-task `query()` sets `outputFormat`. `isComplete()` checks `subtype === 'success' && structured_output.status === 'COMPLETE'`.
- **Fallback**: keep `promiseExtract.ts` (`<promise>([^<]+)</promise>` regex). Used when `structured_output` is absent (schema not set, or `error_max_structured_output_retries`) — degraded but not broken. Keep the prompt-level `<promise>` instruction too: it is the portable convention + a human-readable signal in logs.
- **What this closes**: F33 / Path-4 summarize-risk → structurally eliminated for the primary path. Phase 2.2 acceptance bar ("30 子任务样本 ralph-loop COMPLETE 检测 100% 准确，不被主 summarize 误吞") is now *architecturally guaranteed* on the primary path, not just prompt-engineered.
- Existing v0.1 `src/routing/systemPrompt.ts` + `src/routing/lib/promiseExtract.ts` are **NOT thrown away** — they become the fallback tier. ~30L of new code (schema const + `isComplete()` dual-check), no rewrite.
